1 系统说明

3K用户中心(以下简称UDB) 是专为业务中心提供用户相关服务的技术组件，3K各业务系统可通过本组件实现统一登录及用户关系绑定。
注意：组件遵循OAuth2.0协议标准，开发者在接入之前建议先了解此协议流程。

2 开始开发

2.1 接入概述

接入UDB，业务系统开发人员需要完成以下步骤：

  • UDB已正确添加IP白名单
  • UDB已正确配置回调域名白名单
  • 选择适用的授权模式，依据文档实现业务逻辑

3 授权接入

3.1 获取api_token

api_token是各业务系统全局唯一的接口调用凭证，业务系统在调用UDB各接口时都需要使用api_token，需要妥善保存。api_token有效期为2小时，需定时刷新，重复获取将导致上次获取的api_token失效。

api_token生成及使用说明

  • 建议各业务平台使用中央服务器统一获取和刷新api_token，其它业务逻辑服务器所使用的api_token均来自于该中央服务器，不应该各自去刷新，否则容易造成冲突导致api_token覆盖而影响业务；
  • 目前api_token的有效期通过返回的expire_in来传递，有效期为7200秒。中央服务器需要根据此参数来刷新api_token。在刷新过程中仍可对外继续输出旧api_token，UDB会保证在5分钟内保持新旧api_token都可用。

业务系统可根据UDB分配的client_id和secret来获取api_token

注意，因该API对安全性要求较高，在调用之前业务系统应将服务器IP配置到UDB以便放开白名单

3.2 授权码模式

3.2.1 业务说明

业务系统可以通过3K用户网页授权机制来获取用户基本资源，进而实现业务逻辑；

关于网页授权回调域名说明
目前UDB系统暂不对3K体系外的业务系统使用，为提高安全性，在请求3K用户网页授权之前，业务系统需要将回调域名配置到UDB系统内，开发者中心在上线之前可将域名提供给相应开发人员进行配置。

3.2.2 业务流程

如图所示，授权码模式是OAuth2.0协议中最全也是最标准的模式，总共分为4步：

  • 第一步：跳转至授权页，用户选择是否登录或是否同意授权
  • 第二步：授权完成后，业务系统获取到一次性授权码
  • 第三步：使用授权码获取AccessToken令牌
  • 第四步：使用令牌获取用户资源

3.2.3 唤起授权页

在请求拉取授权页之前，UDB系统会对回调域名作强校验处理，请先确保redirect_uri参数所在域名已在UDB中配置！

注意：scope参数默认为userbase，为保证业务严谨性，此参数不建议留空！

调用方式

参数说明

参数 必填 说明
client_id 是 业务系统唯一标识

state 透传参数，原值返回

3.2.4 换取令牌

该种方式适用于授权码模式，用户在授权完成后，认证服务器会将授权码回调给业务系统服务器，由业务系统使用授权码换取令牌

3.3 密码模式

3.3.1 业务说明

密码模式为OAuth2.0协议中的其中一个标准协议，UDB将密码交由业务系统托管，由业务系统负责收集与通讯，因涉及开放性密码传输，所以必须需要保证使用的业务系统高度可信！

注意：该模式通常应用于内部系统，绝对禁止对外部系统及第三方开放！

3.3.2 业务流程

如图所示，密码模式为OAuth2.0协议中最为直观及最简单但安全度要求最高的方式，可以直接理解为UDB向业务系统提供了一个登录API，业务系统直接通过API通知及校验UDB登录，此种模式相比较网页授权模式区别在于业务系统需要自己定制一个用户登录视图，而网页授权模式中的授权页及登录页均由UDB提供。

注意！！！此种模式要求业务系统绝对高度可信！

业务系统首先需要有自己的账号登录界面，通过登录页获取用户输入的用户名和密码，通过一系列业务系统业务逻辑处理之后，调用以下API进行UDB认证：

3.4 快速授权模式

3.4.1 业务说明

此模式作为密码授权模式的延伸，该模式允许业务系统使用自身的用户唯一标识让用户快速在UDB系统内授权登录，适用于有自己用户唯一标识的业务系统。

注意：该模式通常应用于内部系统，绝对禁止对外部系统及第三方开放！

3.4.2 业务流程

如图所示，快速授权模式为UDB专为业务系统提供的唯一标识快速授权的非OAuth2.0标准模式，业务系统在产生自身用户唯一标识后，允许通过此模式向UDB进行授权（用户无感知），通过UDB认证后直接调取登录服务后返回登录令牌（如果业务系统用户唯一标识在UDB内不存在，则自动调用注册服务优先创建用户）

注意！！！此种模式要求业务系统绝对高度可信！

由于access_token令牌有效期较短，目前支持7200秒存活，业务系统如果需要实现登录状态过期后用户无感知自动续登，可以使用refresh_token进行刷新，refresh_token有效期为30天，刷新后能重新获取一个新的access_token。
